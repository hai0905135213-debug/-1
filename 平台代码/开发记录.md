### 已读取的项目资料

- 读取了 `校园饭搭子平台_完整申报书 人工润色版.docx`。

### 第一阶段产品方向

第一阶段先做 MVP，不做大而全的平台。

MVP 核心闭环：

1. 用户登录并完善基础资料
2. 用户发起一个饭局
3. 其他用户浏览并加入饭局
4. 人数满足后成局
5. 线下就餐后进行简单评价

第一阶段暂缓：

- 复杂课表联动
- 商家抽佣
- PRO 会员
- 大规模推荐算法
- 多学校扩展
- 完整聊天系统

### 初步页面规划

- 首页：展示当前可加入饭局、推荐饭局、快捷发起入口
- 发起饭局页：填写吃什么、时间、地点、人数、预算、聊天模式
- 饭局详情页：查看发起人信息、参与人数、加入/退出饭局
- 我的饭局页：查看我发起的、我加入的、待确认的饭局
- 我的资料页：学生认证、口味偏好、信用分、历史评价

### 数据库升级：内存 Map → SQLite

**决策背景：** 原来的内存 Map 每次重启数据全部丢失，联调时需反复造数据。正式 MySQL 对 MVP 阶段过重（需要独立部署、配权限、管连接池）。

**方案：** SQLite 作为过渡层。零配置、持久化、SQL 语法和 MySQL 一致，后续切 MySQL 只需换驱动和连接串。

**具体变更：**

- 安装依赖：`better-sqlite3`（同步 API，与现有代码风格一致）
- 新增 `backend/src/db.js`：数据库层，负责建表、CRUD、种子数据
  - 5 张表：`users`、`profiles`、`meals`、`meal_participants`、`reviews`
  - 所有返回数据已规范化为 camelCase，前端接口字段无变化
  - 首次启动自动建表 + 写入种子数据，后续重启数据持久保留
  - WAL 模式开启，读性能更好
  - 数据库文件位置：`backend/data/fanfan.db`
- 重构 `backend/src/server.js`：
  - 移除所有 Map 变量和手动 ID 计数器（`nextUserId` 等）
  - 改用 `db.js` 导出的函数进行数据读写
  - 接口逻辑、路由分发、错误处理保持不变
  - health 接口版本升级为 `0.2.0`，新增 `storage: "sqlite"` 字段

**测试结果（全部通过）：**
- ✅ 健康检查（含 storage 字段验证）
- ✅ 登录/注册（新用户创建、老用户复用）
- ✅ 饭局列表（筛选、搜索、仅可加入）
- ✅ 饭局详情（参与者列表、评价列表）
- ✅ 发布饭局（必填校验、自动加入创建者）
- ✅ 加入饭局（状态校验、人数上限、重复加入拦截）
- ✅ 退出饭局（发起人不能退出、正常退出）
- ✅ 饭后评价（权限校验、评分范围校验）

  - 新增 `normalizeRestaurant()` 规范化函数

- `backend/src/server.js`：
  - 新增 3 个路由：`GET /api/restaurants`（列表+筛选）、`GET /api/restaurants/:id`（详情）、`POST /api/restaurants`（新增，需登录）
- `docs/API.md`：新增接口 12-14 餐厅接口文档
- `docs/功能代码对照.md`：新增餐厅相关行

**设计说明：**
- meals 表的 `place` 字段暂不改动，保持自由文本。前端可以从餐厅列表选择填入 place，也可手动输入。
- `avgPrice` 单位为分（2200 = 22元），前端展示时除以 100。
- `tags` 以 JSON 数组存储，方便后续按标签推荐。
- 列表默认按评分降序排列。

**测试结果（8/8 通过）：**
- ✅ 餐厅列表（全量、foodType 筛选、keyword 搜索）
- ✅ 餐厅详情（含 tags 数组解析）
- ✅ 新增餐厅（含 tags 数组、必填校验）
- ✅ 未登录拦截、名称必填校验、404 处理
