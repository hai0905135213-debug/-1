# 项目约定与模式

> 读完这篇你就能理解：这个项目怎么写代码、为什么这么写、后续开发怎么保持一致。

## 1. 命名约定

### 文件命名

| 类型 | 规则 | 示例 |
|---|---|---|
| 源码文件 | 小写 + 短横线（kebab-case）或简短名 | `server.js` `db.js` |
| 文档文件 | 中文描述或数字前缀 | `API.md` `01-架构总览.md` |

### 变量命名

| 场景 | 规则 | 示例 |
|---|---|---|
| JS 变量 | camelCase | `nextUserId` `mealTime` |
| 数据库列名 | snake_case | `meal_time` `creator_id` |
| 数据库表名 | snake_case 复数 | `users` `meal_participants` |
| SQL 查询中的表别名 | 简短缩写 | `mp` (meal_participants), `u` (users) |
| 常量 | UPPER_SNAKE | `PORT` `HOST` `DB_PATH` |

> [!tip] 为什么 JS 用 camelCase、数据库用 snake_case
> 这是两个生态的惯例。JS/JSON API 用 camelCase（`creditScore`），SQL 用 snake_case（`credit_score`）。我们在 normalize 函数里做转换，两边都不别扭。

### 函数命名

| 前缀 | 含义 | 示例 |
|---|---|---|
| `get*` | 查单条，可能返回 null | `getUser(id)` `getMeal(id)` |
| `list*` | 查列表，返回数组 | `listMeals(filters)` `listRestaurants(filters)` |
| `create*` | 新增一条，返回完整对象 | `createUser(data)` `createMeal(data)` |
| `update*` | 更新一条或多字段 | `updateMeal(id, fields)` |
| `upsert*` | 存在则更新，不存在则插入 | `upsertProfile(userId, data)` |
| `handle*` | HTTP 请求处理函数 | `handleLogin` `handleMealList` |
| `is*` / `has*` | 布尔判断 | `isUserJoined(mealId, userId)` |
| `to*` | 数据格式转换 | `toMealSummary()` `toPublicUser()` |
| `send*` | 写 HTTP 响应 | `sendJson()` `sendError()` |
| `normalize*` | 数据库行 → JS 对象 | `normalizeUser()` `normalizeMeal()` |

> [!tip] 字段名转换口诀
> 凡是 `_x` 结尾的，JS 侧把 `_x` 换成大写首字母：`created_at` → `createdAt`，`user_id` → `userId`。
> 凡是 `_xx` 两个单词的，JS 侧直接驼峰：`food_type` → `foodType`，`meal_time` → `mealTime`。

## 3. 代码组织模式

### 函数注释规范

每个导出函数或重要内部函数前都有一行中文注释：

```javascript
// 功能：用户登录。MVP 阶段用昵称/学号创建或复用用户，返回前端调接口所需 token。
async function handleLogin(req, res) { ... }
```

格式：`// 功能：<做什么>。<补充说明>。`

### 参数校验模式

```javascript
// 必填字段校验
const requiredFields = ["title", "mealTime", "place", "maxPeople"];
const missingField = requiredFields.find(
  (field) => body[field] === undefined || body[field] === ""
);
if (missingField) {
  return sendError(res, 400, "FIELD_REQUIRED", `${missingField} 不能为空`);
}

// 范围校验
if (!Number.isInteger(rating) || rating < 1 || rating > 5) {
  return sendError(res, 400, "INVALID_RATING", "评分必须是 1 到 5 的整数");
}

// 格式校验
const mealTime = new Date(body.mealTime);
if (Number.isNaN(mealTime.getTime())) {
  return sendError(res, 400, "INVALID_MEAL_TIME", "mealTime 必须是合法时间");
}
```

校验失败直接 `return sendError(...)`，不继续执行。

### 数据库操作模式

```javascript
// 单条查询
const row = db.prepare("SELECT * FROM xxx WHERE id = ?").get(Number(id));
return normalizeXxx(row);

// 列表查询
return db.prepare(sql).all(...params).map(normalizeXxx);

// 写入
const result = db.prepare("INSERT INTO xxx (...) VALUES (...)").run(...params);
return getXxx(result.lastInsertRowid);
```

## 4. HTTP 状态码约定

| 场景 | 状态码 |
|---|---|
| 查询成功 | 200 |
| 创建成功 | 201 |
| 更新成功 | 200 |
| 请求格式错误/参数缺失 | 400 |
| 未登录 | 401 |
| 无权限操作 | 403 |
| 资源不存在 | 404 |
| 业务冲突（重复加入、已满等） | 409 |
| 服务器内部错误 | 500 |

## 5. 跨域 (CORS) 配置

所有响应统一附加三个 header：

```
Access-Control-Allow-Origin: *
Access-Control-Allow-Methods: GET,POST,PUT,DELETE,OPTIONS
Access-Control-Allow-Headers: Content-Type, Authorization, X-User-Id
```

`OPTIONS` 预检请求返回 204（无内容）。

## 6. 饭局状态流转

```
open ──→ matched ──→ completed
  │                    ↑
  └──→ cancelled       │ (未来功能)
       (未来功能)
```

| 状态 | 含义 | 触发条件 |
|---|---|---|
| `open` | 可加入 | 创建时 / 有人退出 |
| `matched` | 已成局（人数满） | 最后一人加入时 `refreshMealStatus()` |
| `completed` | 已完成（未来） | — |
| `cancelled` | 已取消（未来） | — |

## 7. 聊天模式 (chatMode)

| 值 | 含义 |
|---|---|
| `quiet` | 安静吃饭，少聊天 |
| `balanced` | 正常交流 |
| `talkative` | 可聊天，社交型 |

这是个偏好标签，不影响任何业务逻辑，只在前端展示。
