# 数据库层详解

> 读完这篇你就能理解：表是怎么建的、CRUD 的通用模式、snake_case 到 camelCase 的转换原理、如何新增一张表。

## 1. 文件结构

[db.js](backend/src/db.js) 从上到下分为六个区域：

```
1. 初始化区    — 连接 SQLite、建表
2. normalize区 — 数据格式转换函数
3. 用户区      — users + profiles CRUD
4. 饭局区      — meals + participants CRUD
5. 评价区      — reviews CRUD
6. 餐厅区      — restaurants CRUD
7. 辅助区      — 跨表查询工具
8. 种子数据区  — seedIfEmpty()
```

## 2. 初始化：连接数据库

```javascript
import Database from "better-sqlite3";

const db = new Database(DB_PATH);    // 打开（不存在则创建）
db.pragma("journal_mode = WAL");     // 开启 WAL 模式
db.pragma("foreign_keys = ON");      // 启用外键约束
```

> [!note] 为什么选 better-sqlite3
> 而不是 `sqlite3`（异步API）？因为 `better-sqlite3` 是**同步**的。
> ```javascript
> // better-sqlite3: 简单直接
> const user = db.prepare("SELECT * FROM users WHERE id = ?").get(1);
> 
> // sqlite3: 回调地狱
> db.get("SELECT * FROM users WHERE id = ?", [1], (err, user) => { ... });
> ```
> 同步代码更容易读、调试、维护。Node.js 是单线程的，SQLite 查询通常在微秒级完成，不会阻塞事件循环。

> [!note] WAL 模式是什么
> Write-Ahead Logging。写操作先写日志文件，不直接写主数据库。好处：读和写可以同时进行、崩溃恢复更安全。对于多用户并发读的场景（比如多人同时刷饭局列表）性能更好。

## 3. 建表：CREATE TABLE IF NOT EXISTS

`initTables()` 里每张表都是这个模式：

```sql
CREATE TABLE IF NOT EXISTS users (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,  -- 自增主键
  nickname      TEXT NOT NULL,                       -- 必填
  student_no    TEXT DEFAULT '',                     -- 可选，默认空串
  school        TEXT DEFAULT '示例大学',
  credit_score  INTEGER DEFAULT 100,
  created_at    TEXT NOT NULL                        -- ISO 时间字符串
);
```

> [!tip] 设计约定
> 1. 主键统一用 `INTEGER PRIMARY KEY AUTOINCREMENT`（SQLite 里 INTEGER 主键自增是最快的）
> 2. 时间和 JSON 数组都用 `TEXT` 存（SQLite 没有 native JSON/Date 类型）
> 3. 外键用 `REFERENCES users(id)`，并开启 `foreign_keys = ON`
> 4. 列名统一 `snake_case`（数据库惯例），到 JS 侧转 `camelCase`

**当前全部 6 张表：**

| 表名 | 用途 | 关联 |
|---|---|---|
| `users` | 用户基本信息 | — |
| `profiles` | 用户详细资料 | FK → users.id |
| `meals` | 饭局 | FK → users.id (creator) |
| `meal_participants` | 参与者记录 | FK → meals.id, users.id |
| `reviews` | 评价 | FK → meals.id, users.id ×2 |
| `restaurants` | 餐厅/美食地点 | — |

## 4. normalize：snake_case → camelCase

数据库返回的数据长这样：
```javascript
{ id: 1, food_type: "麻辣香锅", meal_time: "...", max_people: 4, creator_id: 1 }
```

但我们想要这样：
```javascript
{ id: 1, foodType: "麻辣香锅", mealTime: "...", maxPeople: 4, creatorId: 1 }
```

所以每个表都有对应的 normalize 函数：

```javascript
function normalizeMeal(row) {
  if (!row) return null;  // 查不到返回 null，不是 undefined
  return {
    id: row.id,
    title: row.title,
    foodType: row.food_type,     // snake → camel
    mealTime: row.meal_time,
    maxPeople: row.max_people,
    creatorId: row.creator_id,
    // ... 逐字段映射
  };
}
```

> [!warning] 为什么不用自动转换库
> 手写映射看起来笨，但有三个好处：
> 1. **显式**：一眼就知道这个字段从哪个列来
> 2. **安全**：不会不小心暴露不该返回的列
> 3. **可做额外处理**：比如 `tags: JSON.parse(row.tags || "[]")`——JSON 字符串变数组

**normalize 函数对照表：**

| 函数 | 输入 (SQLite row) | 输出 (API JSON) | 特殊处理 |
|---|---|---|---|
| `normalizeUser(row)` | users 行 | user 对象 | — |
| `normalizeMeal(row)` | meals 行 | meal 对象 | — |
| `normalizeParticipant(row)` | 联表查询结果 | participant 对象 | 嵌套 `user` 子对象 |
| `normalizeReview(row)` | reviews 行 | review 对象 | — |
| `normalizeRestaurant(row)` | restaurants 行 | restaurant 对象 | `tags` JSON.parse |

> [!tip] normalizeParticipant 是最复杂的
> 它不只是一个表的数据，而是 `meal_participants LEFT JOIN users` 的结果，所以返回时把用户公开信息（昵称、学校、信用分）嵌在 `user` 字段里：
> ```javascript
> user: row.nickname ? {
>   id: row.user_id,
>   nickname: row.nickname,
>   school: row.school,
>   creditScore: row.credit_score
> } : null
> ```

## 5. CRUD 通用模式

每个表都有 3-4 个操作，遵循同一个模板：

### 查单条：getXxx(id)

```javascript
export function getMeal(id) {
  const row = db.prepare("SELECT * FROM meals WHERE id = ?")
    .get(Number(id));        // .get() 返回第一行或 undefined
  return normalizeMeal(row);  // row 为 undefined 时返回 null
}
```

| 方法 | 返回 | 适用场景 |
|---|---|---|
| `.get(params)` | 单行或 undefined | 按主键查 |
| `.all(params)` | 数组（可能为空） | 列表查询 |
| `.run(params)` | `{ changes, lastInsertRowid }` | INSERT/UPDATE/DELETE |

### 查列表：listXxx(filters)

```javascript
export function listMeals({ status, campus, keyword } = {}) {
  let sql = "SELECT * FROM meals WHERE 1=1";  // 1=1 是个技巧，方便后面拼 AND
  const params = [];

  if (status) {
    sql += " AND status = ?";
    params.push(status);
  }
  if (keyword) {
    sql += " AND (title LIKE ? OR food_type LIKE ? ...)";
    params.push(`%${keyword}%`, `%${keyword}%`, ...);
  }

  sql += " ORDER BY meal_time ASC";
  return db.prepare(sql).all(...params).map(normalizeMeal);
}
```

> [!tip] `WHERE 1=1` 技巧
> 开头的 `WHERE 1=1` 让后续每个筛选条件都可以统一写成 ` AND xxx = ?`，
> 不需要判断"这是第一个条件吗？要不要加 WHERE？"。保持代码干净。

### 新增：createXxx(data)

```javascript
export function createMeal(data) {
  const result = db.prepare(`
    INSERT INTO meals (...columns...)
    VALUES (?, ?, ?, ?, ...)
  `).run(...values);

  // 拿自增 ID 回查完整行
  return getMeal(result.lastInsertRowid);
}
```

模式：INSERT → 拿 `lastInsertRowid` → 回调查询返回完整对象（经过 normalize）。

> [!warning] 为什么 INSERT 后还要 SELECT
> 因为 normalize 函数的字段可能包含默认值（如 `createdAt`、`status`），INSERT 时我们不填全。回查一次能保证返回给前端的数据是完整的。

### 更新：updateXxx(id, fields)

```javascript
export function updateMeal(id, fields) {
  const sets = [];
  const params = [];
  for (const [key, value] of Object.entries(fields)) {
    sets.push(`${key} = ?`);
    params.push(value);
  }
  params.push(Number(id));
  db.prepare(`UPDATE meals SET ${sets.join(", ")} WHERE id = ?`)
    .run(...params);
  return getMeal(id);
}
```

> [!note] 注意：fields 的 key 必须是 snake_case
> `updateMeal(id, { status: "completed", updated_at: now })` ✅
> `updateMeal(id, { status: "completed", updatedAt: now })` ❌（camelCase 不匹配列名）

## 6. 种子数据：seedIfEmpty()

```javascript
export function seedIfEmpty() {
  const count = db.prepare("SELECT COUNT(*) as cnt FROM users").get();
  if (count.cnt > 0) return;  // 已有数据，跳过
  // ... 插入示例数据
}
```

`server.js` 启动时调用，逻辑很简单：如果 users 表已经有数据就跳过，否则插入预设数据。

**种子数据包含：**
- 2 个用户：小林、阿晴
- 3 个饭局：麻辣香锅（open）、米线（matched）、轻食（open）
- 参与者记录：小林创建了饭局 1 和 3，阿晴创建了饭局 2，小林加入了饭局 2
- 6 个餐厅：覆盖各类美食

> [!tip] 怎么重置数据
> 删掉 `backend/data/fanfan.db` 文件再启动，种子数据会自动重建。

## 7. 如何新增一张表（实践指南）

假设要加一张 `reports`（举报记录）表，步骤是：

1. **initTables()** 加 `CREATE TABLE IF NOT EXISTS reports (...)`
2. 写 **normalizeReport(row)** 转换函数
3. 写 CRUD：`getReport()` `listReports()` `createReport()`
4. seedIfEmpty() 里加示例数据（可选）
5. 删掉 `data/fanfan.db` 重新生成

完全遵循现有表的模式即可，不会影响其他代码。

## 8. better-sqlite3 vs MySQL 迁移备忘

| | better-sqlite3 | mysql2 |
|---|---|---|
| 连接 | `new Database(path)` | `mysql.createPool(config)` |
| 查询 | `db.prepare(sql).all(params)` | `await pool.execute(sql, params)` |
| 自增ID | `result.lastInsertRowid` | `result.insertId` |
| 占位符 | `?` | `?` (一样!) |
| AUTOINCREMENT | `INTEGER PRIMARY KEY AUTOINCREMENT` | `INT AUTO_INCREMENT` |
| INSERT OR UPDATE | `ON CONFLICT DO UPDATE` | `ON DUPLICATE KEY UPDATE` |

**迁移成本很低**——主要是驱动 API 差异，SQL 语句大量复用。
